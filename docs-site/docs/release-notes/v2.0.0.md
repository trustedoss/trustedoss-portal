# v2.0.0 — General Availability

**Release date:** 2026-05-09

The first general-availability release of TrustedOSS Portal. Promotes `2.0.0-rc.1` and absorbs the post-rc cleanup wave (PRs #28 ~ #31).

:::tip Upgrading from rc.1?
Migrations apply automatically and there are no new mandatory `.env` variables. Pull, then `alembic upgrade head` inside the backend container — see [Upgrading from rc.1](#upgrading-from-rc1).
:::

## Added

### Authentication UX (PR #28)

- **`/forgot-password`** is now wired to the backend — submit an email and receive a 24-hour, single-use reset link.
- **`/reset-password?token=…`** is a new page that accepts the token from the email and sets the new password.
- **`i18next-parser` drift gate** lands in CI — adding an EN key without the matching KO key fails the PR.
- **OAuth on `/login`** — GitHub and Google buttons, automatic personal-team provisioning on first sign-in, identity matching by `(provider, provider_user_id)`, seven-error-code i18n mapping, and `redirect_after` pass-through.

> See also: [Authentication & profile](../user-guide/auth-and-profile.md).

### `/integrations` page (PR #28)

- Self-service API key list, create, and revoke flows.
- One-time plaintext reveal modal with copy-to-clipboard and a hard warning that the full key is shown only once.
- Webhook URL info: `https://<host>/v1/webhooks/github` (HMAC `X-Hub-Signature-256`) and `https://<host>/v1/webhooks/gitlab` (token header).

> See also: [Integrations](../user-guide/integrations.md).

### Backup automation (PR #29)

- **Daily Celery Beat backup** at 00:00 UTC with **7-day automatic retention** for `auto-*` backups.
- **`/admin/backup` UI** with manual trigger, streaming download, type-`restore` Upload+Restore (with `X-Confirm-Restore` header), and delete (auto-* protected).
- `useScanWebSocket` immediate reconnect on tab focus — scan progress updates resume without waiting for the next reconnect tick.

> See also: [Backup & restore](../admin-guide/backup-and-restore.md).

### Locust load harness (PR #30)

- `tests/load/` plus `docker-compose.load.yml` for staging-only **p95 < 1 s SLO** validation.
- Not wired to CI — run on demand against a staging environment.

### SCA self-scan (PR #30)

- Nightly `sca-self.yml` workflow generates a CycloneDX SBOM with `cdxgen`, scans with Trivy, and **opens / closes a GitHub issue automatically** when CRITICAL vulnerabilities are detected.
- The portal now eats its own dog food.

### API Keys + Webhooks test coverage (PR #31)

- **125 tests** (52 unit + 30 + 23 + 20 integration) bringing `services/api_key_service.py` from **0 % to 88.24 %** line coverage.
- Adversarial-input parametrize matrices on every webhook URL, signature, and SPDX-expression parser.

## Changed

### SAST CI is now HARD FAIL (PR #30)

- **`bandit`** blocks on **High+**.
- **`semgrep`** blocks on **ERROR**.
- **Trivy image-scan** blocks on **CRITICAL**.
- HIGH Trivy findings remain advisory until the Phase 8 worker-image refresh.

## Fixed

- `tasks.backup` lazy `scripts/` resolution — the module now imports cleanly inside `/app/tasks/` containers; the previous `parents[3]` lookup exceeded container path depth (PR #29).
- `.semgrepignore` and inline justifications for **21** semgrep ERROR findings — none represent real risk; documented in `.semgrepignore` and per-line `# nosemgrep` comments (PR #30).
- Pinned `setuptools<78` in the SAST workflow so `pkg_resources` stays importable for the `opentelemetry` packages transitively pulled in by `semgrep` (PR #30).

## Upgrading from rc.1

The upgrade is straightforward — there are **no new mandatory environment variables** in 2.0.0 and migrations apply forward-only.

```bash
cd /opt/trustedoss-portal
git pull origin main

# pull the new images
docker-compose -f docker-compose.yml pull

# apply migrations (no-op if you are already at head)
docker-compose -f docker-compose.yml exec backend alembic upgrade head

# restart with the new images
docker-compose -f docker-compose.yml up -d
```

The migration window is < 60 seconds on a small install. Existing JWT sessions remain valid; users do not need to sign in again.

If you want to use the new daily backup, no action is required — it is enabled by default with `BACKUP_DAILY_ENABLED=true`.
