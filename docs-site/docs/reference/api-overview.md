# API overview

The portal exposes a REST API rooted at `/api/v1`. The full OpenAPI 3.1 schema is generated by FastAPI and served live at `https://<your-portal>/api/docs` (Swagger UI) and `/api/redoc` (Redoc). This page is a high-level orientation.

:::note Audience
Engineers integrating with the portal — CI runners, partner tooling, custom dashboards. Familiarity with HTTP, JSON, and OAuth-style bearer tokens.
:::

## Base URL

```
https://<your-portal>/api/v1
```

Trailing slashes are normalized — both `/projects` and `/projects/` work.

## Authentication

Two auth schemes are accepted on every protected endpoint:

### Bearer JWT (interactive sessions)

```http
Authorization: Bearer <access_token>
```

Issued by `POST /api/v1/auth/login`. 30-minute lifetime. Refresh via the rotation cookie returned at login.

### API key (machine clients)

```http
Authorization: ApiKey tos_<prefix>_<secret>
```

`Authorization: Bearer <key>` is also accepted for convenience. See [API keys](../admin-guide/api-keys.md).

### Anonymous endpoints

Only the webhook receivers (`/webhooks/github`, `/webhooks/gitlab`) and `GET /health` are unauthenticated. Webhooks rely on HMAC / token verification from the request body / header.

## Errors — RFC 7807

All 4xx and 5xx responses carry `Content-Type: application/problem+json` with this shape:

```json
{
  "type":     "https://trustedoss.io/problems/forbidden",
  "title":    "Forbidden",
  "status":   403,
  "detail":   "API key 'tos_a1b2c3d4_…' lacks required action 'scan:trigger'.",
  "instance": "/api/v1/projects/01H…/scans"
}
```

Domain extensions are `snake_case` and modelled in the OpenAPI schema. Two well-known examples:

| Type URI | Status | Triggered by |
|---|---|---|
| `…/last-super-admin` | 409 | Demoting the last super-admin. |
| `…/disk-pressure` | 503 | New scan rejected because disk is above hard limit. |

## Pagination

List endpoints accept:

| Query param | Default | Description |
|---|---|---|
| `page` | `1` | 1-based page number. |
| `size` | `50` | Page size. Max 200. |
| `sort` | endpoint-specific | Comma-separated `field` or `-field` (descending). |

Response envelope:

```json
{
  "items": [ … ],
  "page": 1,
  "size": 50,
  "total": 1273,
  "next": "/api/v1/projects?page=2&size=50",
  "prev": null
}
```

`next` is `null` on the last page. `total` is omitted on very large tables when an exact count is too costly; a `has_more: true` boolean takes its place.

## Idempotency

Mutating endpoints accept `Idempotency-Key: <uuid>` and store the response for 24 hours. Re-sending the same key returns the cached response without re-executing the side effects. Use this for client retries on flaky networks.

## WebSockets

The portal exposes one WebSocket endpoint:

```
WSS  /api/v1/scans/{id}/progress
```

Authenticate via the same JWT (passed as `?token=<jwt>` query string — browsers cannot set headers on WebSocket upgrade). Messages:

```json
{ "scan_id": "…", "stage": "resolving_vulnerabilities", "progress": 0.62, "message": "…", "ts": "…" }
```

Reconnect with exponential backoff. Each reconnect is given the latest stage so the UI converges quickly.

## OpenAPI download

```bash
curl -sS https://trustedoss.example.com/api/openapi.json > openapi.json
```

The schema is regenerated at startup. Pin against a release tag if you generate clients (`openapi-generator-cli`, `openapi-typescript`).

## Rate limits

- Login (`/auth/login`): IP-keyed 5/minute. 429 with `Retry-After: 60`.
- Most other endpoints: user-keyed 600/minute (effectively no limit for normal use).
- API keys are subject to the same per-user policy.

The headers on every response include:

```
X-RateLimit-Limit:     600
X-RateLimit-Remaining: 599
X-RateLimit-Reset:     1715269200
```

Pause and back off when `Remaining` is 0.

## Observability

Set `X-Request-ID` on outbound calls; the portal echoes it in the response and logs it on every line for that request. Without the header, the portal generates a UUIDv7 and returns it.

## Versioning

The path includes `/v1`. Breaking changes go to `/v2`. Within `/v1`:

- New optional fields on responses are not breaking.
- New required fields on requests are gated behind a new endpoint or behind a feature header.
