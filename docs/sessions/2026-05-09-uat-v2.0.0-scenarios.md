## 시나리오 16: 비밀번호 찾기 + 재설정 (PR #28)

> **계정**: Anonymous (로그인 안 된 상태)
> **사전 조건**: SMTP 서버 동작 (`SMTP_HOST` / `SMTP_USER` / `SMTP_PASSWORD` 설정) 또는 backend 로그에서 메일 본문 확인 가능

### 16-1. 정상 흐름 — 토큰 발급부터 새 비밀번호 로그인까지

1. `/login` 접속 → 하단의 **Forgot password?** 링크 클릭
2. `/forgot-password` 페이지 → Email 입력 (`dev@trustedoss.dev`) → **Send reset link** 클릭
3. 화면에 균일한 success view 표시 (`If an account with that email exists, a reset link has been sent.`)
4. SMTP 메일함 또는 backend 로그에서 메일 본문 확인 → `https://<host>/reset-password?token=<32자 이상 랜덤>` 링크 추출
5. 링크 클릭 → `/reset-password?token=...` 페이지 진입
6. New password (`NewPasswordTest2026!`) + Confirm password 입력 → **Reset** 클릭
7. `/login` 으로 리다이렉트 → 새 비밀번호로 로그인 시도

**기대 결과**
- 3번 단계에서 항상 동일한 success view 노출 (이메일 존재 여부 누설 X)
- 메일 본문에 토큰 링크 + 만료시각(24h) 안내 포함
- 6번 클릭 후 200 응답 + 토스트 "Password reset successful"
- 7번 로그인 성공
- 기존 비밀번호 (`TrustedDev2026!`)로는 로그인 실패 (`Invalid credentials`)

**스크린샷**: `![](./img/uat-16-1.png)`

---

### 16-2. Anti-enumeration — 존재하지 않는 이메일

1. `/forgot-password` → Email 입력 (`nonexistent@example.com`) → **Send reset link** 클릭

**기대 결과**
- 16-1과 **완전히 동일한** success view ("If an account with that email exists ...")
- 응답 시간도 인지 가능한 차이 없음 (timing attack 방지 — 백엔드 sleep 또는 동일 코드 경로)
- backend 로그에는 메일 발송 시도 없음 (단, 400/404 등의 에러 응답도 X)

**스크린샷**: `![](./img/uat-16-2.png)`

---

### 16-3. 토큰 만료 / 재사용 차단

1. 16-1로 받은 토큰 링크를 **재사용** 시도 (이미 비밀번호를 한 번 변경한 토큰)
2. 별도로, 24시간 지난 토큰 (DB에서 `created_at`을 25시간 전으로 수정 또는 새 발급 후 25h 대기) 으로 시도

**기대 결과**
- 1번: `/reset-password?token=<used>` 진입 시 명시적 안내 ("This reset link has already been used")
- 2번: `/reset-password?token=<expired>` 진입 시 명시적 안내 ("This reset link has expired")
- 두 경우 모두 새 비밀번호 입력 폼 비활성 + "Request a new link" 버튼 노출
- `/login` 페이지 즉시 리다이렉트 X (사용자가 상황을 인지할 수 있어야 함)

**스크린샷**: `![](./img/uat-16-3.png)`

---

### 16-4. 토큰 누락 (직접 URL 접근)

1. `/reset-password` (쿼리 파라미터 없이) 직접 접속

**기대 결과**
- 명시적 안내 UI ("Reset token is missing or invalid")
- "Request a new link" 버튼 → `/forgot-password` 로 이동
- 익명 폼 노출 X (token 없이 비밀번호 변경 시도 차단)

**스크린샷**: `![](./img/uat-16-4.png)`

## 시나리오 17: OAuth 로그인 — GitHub 신규 사용자 자동 가입 (PR #28)

> **계정**: Anonymous + GitHub 본인 계정 (포털에 미등록 상태)
> **사전 조건**: `.env`에 `GITHUB_CLIENT_ID` / `GITHUB_CLIENT_SECRET` 설정. GitHub OAuth App callback URL이 `http://localhost:8000/v1/auth/github/callback` 으로 설정되어 있을 것.

### 17-1. 신규 GitHub 계정 첫 로그인 → 자동 가입

1. 포털에서 해당 GitHub email로 등록된 사용자가 없는지 확인 (Super Admin → `/admin/users` 검색)
2. 로그아웃 상태에서 `/login` 진입 → **Continue with GitHub** 버튼 클릭
3. GitHub 동의 화면 → **Authorize** 클릭
4. `http://localhost:8000/v1/auth/github/callback?code=...&state=...` 콜백
5. 백엔드가 사용자 자동 생성 + JWT 발급 → 프론트로 리다이렉트
6. `/` 대시보드 진입 (자동 로그인 완료)

**기대 결과**
- 3번 동의 화면이 GitHub UI에 표시 (email 권한 요청 포함)
- 5번 후 `/admin/users` 에서 새 사용자 확인 가능 (provider=`github`, email=GitHub 기본 이메일)
- 자동 생성된 사용자는 개인 Team에 소속 (조직 단위 배포라면 default team, demo SaaS면 user-{id} team)
- `/profile` 진입 시 Connected Accounts에 GitHub 표시 (시나리오 19에서 상세 검증)

**스크린샷**: `![](./img/uat-17-1.png)`

---

### 17-2. 7개 OAuth 에러 코드 i18n 매핑 검증

| 에러 코드 | 트리거 방법 | 기대 메시지 (EN) | 기대 메시지 (KO) |
|----------|-----------|----------------|----------------|
| `oauth_user_denied` | GitHub 동의 화면에서 **Cancel** 클릭 | "You cancelled GitHub sign-in." | "GitHub 로그인을 취소했습니다." |
| `oauth_state_mismatch` | DevTools로 콜백 URL의 `state` 파라미터를 임의 변경 | "Sign-in state mismatch. Please try again." | "로그인 상태 검증 실패. 다시 시도해 주세요." |
| `oauth_provider_unavailable` | `.env`에서 `GITHUB_CLIENT_ID` 비우고 재시작 → GitHub 버튼 클릭 | "GitHub sign-in is currently unavailable." | "GitHub 로그인이 현재 비활성 상태입니다." |
| `oauth_email_unverified` | GitHub primary email을 unverified로 만든 후 시도 | "Please verify your GitHub email first." | "GitHub 이메일을 먼저 인증해 주세요." |
| `oauth_email_missing` | scope에서 email 권한 거부 | "GitHub did not return an email address." | "GitHub에서 이메일 주소를 받지 못했습니다." |
| `oauth_account_collision` | 같은 email의 password user가 이미 존재할 때 | "An account with this email already exists. Please sign in with password and link GitHub from /profile." | "이 이메일로 가입된 계정이 있습니다. 비밀번호로 로그인 후 /profile에서 GitHub를 연결해 주세요." |
| `oauth_internal_error` | backend OAuth handler에 임의 예외 주입 (테스트 fixture) | "An unexpected error occurred during sign-in." | "로그인 중 예기치 않은 오류가 발생했습니다." |

각 행마다:

1. 해당 트리거 적용 → GitHub 버튼 클릭 → 콜백 도달
2. 프론트 URL이 `/login?error=<code>`로 리다이렉트되는지 확인
3. `/login` 화면 상단에 i18n 메시지가 EN/KO 모두 위 표대로 노출되는지 확인

**기대 결과**
- 7개 모든 에러 코드가 명시적 사용자-친화 메시지로 매핑
- 메시지에 stacktrace / 내부 식별자 / 토큰 노출 X
- KO 모드에서도 영문 fallback 없이 한국어 표시

**스크린샷**: `![](./img/uat-17-2.png)`

## 시나리오 18: OAuth 로그인 — Google 기존 사용자 식별자 매칭 (PR #28)

> **계정**: Developer (`dev@trustedoss.dev`) + Google 본인 계정 (email은 dev 계정과 동일하다고 가정)
> **사전 조건**: `.env`에 `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` 설정.

### 18-1. 동일 email Google 계정 → 기존 user에 link

1. 포털에 password user (`dev@trustedoss.dev`)가 존재 + Google identity는 아직 없음 상태 확인
2. 로그아웃 → `/login` → **Continue with Google** 클릭
3. Google 동의 → 콜백
4. backend가 email 매칭 → 기존 user에 google identity 추가 (link) → 자동 로그인
5. `/profile` 진입 → Connected Accounts에 Google 표시 + 기존 password 계정 그대로

**기대 결과**
- 새 user 생성 X (사용자 ID는 그대로 유지) — DB `users` 테이블 row count 변동 없음
- `oauth_identities` 테이블에 새 row 추가 (`provider=google`, `user_id=<existing dev user id>`)
- 자동 로그인 후 시나리오 19로 unlink 가능 (현재 password 있으므로 unlink OK)
- 단, 기존 정책상 collision 차단 모드라면 17-2의 `oauth_account_collision` 에러 노출 — 정책 모드를 미리 확인하고 시나리오 분기

**스크린샷**: `![](./img/uat-18-1.png)`

---

### 18-2. 다른 email Google 계정 → 신규 user 생성

1. 포털에 등록되지 않은 별도의 Google 계정으로 시도
2. `/login` → **Continue with Google** → 동의 → 콜백

**기대 결과**
- 17-1과 동일하게 신규 user 생성 + 자동 로그인
- `/admin/users` 에서 provider=`google` 확인 가능
- 개인 Team 자동 생성 (demo SaaS 모드)

**스크린샷**: `![](./img/uat-18-2.png)`

## 시나리오 19: /profile Connected Accounts + Unlink (PR #33)

> **계정**: Developer (`dev@trustedoss.dev`)
> **사전 조건**: 시나리오 18-1을 먼저 수행하여 password + Google identity가 모두 있는 상태

### 19-1. /profile Connected Accounts 목록 확인

1. 로그인 후 우상단 사용자 메뉴 → **Profile** 클릭 → `/profile` 진입
2. Connected Accounts 섹션 확인

**기대 결과**
- 표시 컬럼: Provider (github / google) / Linked at (timestamp) / Unlink 버튼
- 현재 연결된 identity 수 표시 (예: "2 connected accounts" — google + github 모두 연결한 경우)
- password 사용 가능 여부 표시 ("Password sign-in is enabled" 안내 텍스트)

**스크린샷**: `![](./img/uat-19-1.png)`

---

### 19-2. password 있음 + 단일 OAuth → unlink 가능

1. 19-1 상태에서 Google 옆 **Unlink** 버튼 클릭
2. 확인 모달 → **Unlink** 클릭

**기대 결과**
- 200 응답 + 토스트 "Google account unlinked"
- Connected Accounts 목록에서 Google 제거
- 다음에 Google OAuth로 로그인 시도 → 17-2의 collision 또는 신규 가입 흐름으로 분기 (정책에 따름)
- password 로그인은 계속 가능

**스크린샷**: `![](./img/uat-19-2.png)`

---

### 19-3. password 없음 + 단일 OAuth → unlink 차단 (409)

> **사전 조건**: OAuth-only 사용자 준비 — 시나리오 17-1에서 만든 GitHub-only 신규 사용자로 로그인 (해당 user는 `hashed_password=NULL`)

1. 해당 user로 `/profile` 진입 → Connected Accounts에 GitHub 1개만 표시
2. GitHub 옆 **Unlink** 버튼 클릭 → 확인 모달 → **Unlink**

**기대 결과**
- 409 Conflict + RFC 7807 problem+json 응답
  - `title`: "Cannot unlink last sign-in method"
  - `detail`: "Set a password first or link another provider before unlinking."
- 프론트는 alert 또는 inline error로 위 메시지 노출
- Connected Accounts 목록 변경 없음 (GitHub identity 그대로)
- "Set a password" 또는 "Link another provider" 액션 링크 노출 (UX 가이드)

**스크린샷**: `![](./img/uat-19-3.png)`

---

### 19-4. password 없음 + 2개 OAuth → 1개 unlink 가능

1. 19-3의 GitHub-only user에 추가로 Google identity link 수행 (별도 흐름: `/profile`에서 "Link Google" 버튼 또는 OAuth re-flow)
2. `/profile` Connected Accounts에 GitHub + Google 2개 모두 표시 확인
3. Google 옆 **Unlink** 클릭 → 확인 → **Unlink**

**기대 결과**
- 200 응답 (다른 OAuth가 fallback로 남아있어 unlink 가능)
- Connected Accounts에서 Google 제거 → GitHub만 남음
- 이 시점에서 GitHub도 unlink 시도하면 19-3과 동일한 409 발생 (재현 가능)

**스크린샷**: `![](./img/uat-19-4.png)`

## 시나리오 20: /integrations API Key 생성 + 평문 1회 노출 + 폐기 (PR #28)

> **계정**: Developer 또는 Team Admin (`dev@trustedoss.dev`)
> **사전 조건**: 시드 프로젝트 (`my-nodejs-app`) 존재

### 20-1. API Key 생성 + 평문 1회 노출

1. 사이드바 **Integrations** 클릭 → `/integrations` 진입
2. **API Keys** 탭 → **Create API Key** 버튼 클릭
3. 모달에서 입력:
   - Name: `ci-pipeline-test`
   - Scope: `project` 선택 → 프로젝트 드롭다운에서 `my-nodejs-app` 선택
   - Expiry: `90 days` 선택
4. **Create** 클릭

**기대 결과**
- 모달에 평문 키 1회 노출: `tsoss_<random 40+ char>` 형태
- 평문 옆 **Copy** 버튼 → 클릭 시 클립보드 복사 + 토스트
- 경고 문구: "This key will only be shown once. Save it securely now."
- 모달 하단 **Done** 또는 X 버튼

**스크린샷**: `![](./img/uat-20-1.png)`

---

### 20-2. 모달 닫은 후 재진입 — 평문 안 보임

1. 20-1 모달 **Done** 클릭하여 닫기
2. API Keys 목록에서 방금 생성한 `ci-pipeline-test` 행 확인
3. 행 클릭 또는 행 우측 **View** 버튼 클릭

**기대 결과**
- 행 표시 형식: `tsoss_a1b2c3d4...XYZ4` (앞 8자리 + `...` + 마지막 4자리만 노출)
- 상세 드로어/모달에서도 평문 전체 노출 X
- DB에는 hashed key (bcrypt 또는 SHA256)만 저장 — 어떤 경로로도 복원 불가
- "Lost the key? Revoke and create a new one." 안내

**스크린샷**: `![](./img/uat-20-2.png)`

---

### 20-3. Revoke (폐기)

1. API Keys 목록에서 `ci-pipeline-test` 행의 **Revoke** 버튼 클릭
2. 확인 모달에서 **Revoke** 클릭
3. 별도 터미널에서 폐기된 키로 API 호출 시도:
   ```bash
   curl -H "X-API-Key: <copied plaintext from 20-1>" \
        http://localhost:8000/v1/projects/<my-nodejs-app id>
   ```

**기대 결과**
- 2번 후 행 상태가 `Revoked` 배지로 변경 (또는 행 회색 처리)
- 3번 호출은 **401 Unauthorized** + RFC 7807 problem
  - `title`: "Invalid API key"
- DB의 키 row는 soft-delete 또는 `revoked_at` 타임스탬프 세팅
- 감사 로그 (`/admin/audit`)에 `api_key_revoked` 액션 기록

**스크린샷**: `![](./img/uat-20-3.png)`

### 27-4. backend 에러 응답의 RFC 7807 problem locale

1. KO 모드에서 의도적인 에러 트리거 (예: 시나리오 19-3의 unlink 차단 409, 시나리오 22-4의 413)
2. DevTools Network 탭에서 응답 body 확인 (`application/problem+json`)

**기대 결과**
- 응답 헤더 `Content-Type: application/problem+json`
- 응답 body의 `title` / `detail` 이 사용자 locale에 맞게 EN 또는 KO
  - 백엔드는 `Accept-Language` 헤더 또는 JWT claim의 `locale` 값으로 결정
  - frontend는 이를 그대로 alert/toast/inline error에 노출
- `type` (URI), `status`, `instance`는 locale 무관 (URI / 숫자)
- frontend가 추가 i18n 변환 없이 backend 메시지를 신뢰해도 사용자에게 자연스러움

## 알려진 제한 사항

- **OAuth 시나리오 17·18**: 실제 GitHub/Google OAuth App 클라이언트 ID/Secret이 필요합니다. CI / 일반 dev 환경에서는 mock OAuth provider (예: `oauth-mock-server`) 사용 권고.
- **GCP 시나리오 26**: 실제 비용 발생 (Cloud SQL 시간당 약 $0.05+, Cloud Run 호출당). 26-5 destroy 누락 시 결제 누적 — 반드시 자동화된 cleanup hook 또는 일정 알림 설정 권고.
- **시나리오 22-3 Restore 검증**: 실제 데이터 교체가 일어나므로 **테스트 환경에서만** 수행. 프로덕션 데이터 백업 파일을 잘못 업로드하면 복구 불가.
- **시나리오 24-4 SMTP 검증**: 실제 SMTP 서버가 없으면 backend 로그의 `notification.email.skipped` 라인으로 검증. SMTP 연동 통합 테스트는 별도 환경 필요.
- **시나리오 25 WebSocket**: 실제 스캔 실행 (cdxgen / ORT / Trivy)이 worker 이미지에 도구가 설치되어 있어야 진행 이벤트가 발생. 시드 데이터만으로는 WS 이벤트가 생성되지 않을 수 있음 — 별도 스크립트 (`scripts/replay_ws_events.py`) 사용 권고.
